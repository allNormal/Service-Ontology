Abstract

This Ontology is created to support AuditBox.

Introduction back to ToC

Through the use of this ontology, we are able to construct a data model that visualizes AuditBox's configuration while concurrently establishing links with the relevant plan. There are four main classes in the new developed service ontology:

  1. System: The primary function of the System class is to serve as an anchor node for the model, meaning that everything will be connected to the system.
  2. AuditBoxService: The main function of the Service class is to create a REST API that should be available for the system. This includes all the data necessary for the REST API to work. There are currently two subclasses within the Service class, each differing based on their use cases:
    1. Acquire: The Acquire category can be further subdivided into Fetch and Receive. The distinction between the two lies in their methods of data acquisition. Fetch retrieves data from external sources, while Receive waits for incoming data from external sources. An additional subclass, TimerFetch, is available for scheduled recurring data collection.
    2. Query: This class enables querying of available data. For scenarios where filtering based on user input is necessary, ParameterizedQuery, a subclass of Query, offers this capability.
  3. ServiceFunction: This class's purpose is to help Acquire by defining the subsequent steps following the creation of an endpoint. Two subclasses have been defined in the service function called Transformation and Validation. Transformation outlines data mappings for the acquired information, while Validation focuses on data validation.
  4. Parameter: This class assists ParameterizedQuery by defining query parameters. These parameters enable mapping to corresponding counterparts within the query itself.
